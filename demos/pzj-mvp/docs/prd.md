# CodeAgent CLI MVP PRD

产品名称：`GoCode`  
版本：`MVP v0.1`  
产品形态：`本地 CLI + LLM + 工具执行器 + 文件系统访问`

## 1. 产品目标
写一个go语言版的coding agent
让开发者通过自然语言，在本地代码仓库内完成以下闭环：

1. 理解代码库
2. 生成或修改代码
3. 管理文件
4. 调用简单工具或命令
5. 校验结果
6. 输出变更摘要并支持继续迭代

闭环成立的标准是：用户给出任务后，系统能自主完成“读代码 -> 找位置 -> 改代码 -> 跑验证 -> 返回结果”，而不是只生成一段代码文本。

## 2. 目标用户

- 个人开发者
- 小团队工程师
- 需要快速搭建、修改、重构项目的技术人员
- 更偏向终端工作流、希望减少上下文切换的用户

## 3. MVP 范围

### 必须做的能力

- 自然语言输入任务
- 代码生成
- 代码修改
- 文件创建、删除、重命名、移动
- 代码搜索与文件读取
- 简单工具调用
- 命令执行结果读取
- 基于结果继续修复或补充
- 变更展示与操作确认

### 明确不做的能力

- 多 Agent 协作
- 云端远程执行
- 复杂 IDE 插件生态
- 长期记忆和团队知识库
- 自动部署、自动发版
- 图形化界面优先版本

## 4. 核心场景

- “帮我在 `src/api` 下新增一个用户模块，包含路由、service 和测试。”
- “把当前登录逻辑改成 JWT，并保持原有接口不变。”
- “删除没用到的工具函数并修复引用。”
- “运行测试，定位失败原因并修复。”
- “帮我批量重命名目录下的文件，并更新 import。”

## 5. 核心用户流程

1. 用户输入自然语言任务。
2. 系统分析任务，决定需要读取哪些文件、搜索哪些内容、是否需要调用命令。
3. 系统调用工具获取上下文。
4. 系统生成执行计划并开始操作代码。
5. 系统执行文件修改、创建、删除或命令运行。
6. 系统读取工具结果，判断是否需要继续迭代修复。
7. 系统输出最终结果，包括变更摘要、文件列表、命令结果、下一步建议。
8. 用户可继续追问、要求回退、要求补充修改。

## 6. 功能需求

| 编号 | 功能 | 优先级 | 说明 | 验收标准 |
|---|---|---|---|---|
| FR-01 | 自然语言任务输入 | P0 | 用户可直接描述需求 | 能识别“新增/修改/删除/运行/重构”等任务 |
| FR-02 | 项目上下文获取 | P0 | 支持列目录、读文件、全文搜索 | 能在指定仓库内找到相关文件与代码片段 |
| FR-03 | 代码生成 | P0 | 可新增单文件或多文件代码 | 能按任务创建可保存的代码文件 |
| FR-04 | 代码修改 | P0 | 能基于现有代码做定点修改 | 修改后文件结构不被破坏，保留未改部分 |
| FR-05 | 文件管理 | P0 | 创建、删除、重命名、移动 | 文件操作成功并可回显结果 |
| FR-06 | 简单工具调用 | P0 | 支持执行受控命令 | 能运行如测试、构建、格式化等命令并返回输出 |
| FR-07 | 校验与自修复 | P0 | 根据命令输出继续修复一次或多次 | 测试失败时能读取报错并尝试再次修改 |
| FR-08 | 变更展示 | P0 | 输出改动文件、摘要、命令结果 | 用户能清楚看到本轮发生了什么 |
| FR-09 | 风险确认 | P0 | 删除文件、覆盖文件、危险命令需确认 | 高风险操作必须经过用户确认 |
| FR-10 | 会话连续性 | P1 | 保留本轮任务上下文 | 用户追问时无需重复完整背景 |
| FR-11 | Git 辅助 | P1 | 查看 diff、生成提交说明 | 能输出本轮变更的 Git diff 摘要 |

## 7. MVP 工具清单

建议最小工具集如下：

- `list_dir(path)`：列目录
- `read_file(path)`：读文件
- `search_text(query, path)`：全文搜索
- `write_file(path, content)`：写文件
- `apply_patch(path, patch)`：精确修改文件
- `make_dir(path)`：建目录
- `move_path(from, to)`：移动或重命名
- `delete_path(path)`：删除文件或目录
- `run_command(cmd, cwd)`：执行简单命令
- `git_diff()`：查看本轮变更，MVP 可选

这里最关键的是 `apply_patch`，否则“修改代码”会退化成整文件重写，稳定性很差。

## 8. 交互设计

推荐先做 CLI：

- 默认输入：直接输入自然语言任务
- `/plan`：显示本轮执行计划
- `/diff`：显示本轮改动摘要
- `/files`：显示涉及文件
- `/undo`：回滚本轮变更，MVP 可选
- `/exit`：退出会话

输出需要固定包含：

- 当前理解的任务
- 正在调用的工具
- 涉及的文件
- 代码改动摘要
- 命令执行结果
- 是否完成或仍需用户决策

## 9. 产品原则

- 默认以“可控”优先于“全自动”
- 默认以“补丁修改”优先于“整文件重写”
- 默认以“仓库内显式可见操作”优先于“隐式推断”
- 默认先读上下文，再改代码
- 默认危险操作需确认

## 10. 系统模块

| 模块 | 职责 |
|---|---|
| CLI 层 | 接收用户输入、展示结果、处理确认 |
| Agent 编排层 | 解析任务、规划步骤、决定调用哪些工具 |
| Context Manager | 管理上下文窗口，决定读哪些文件与搜索哪些内容 |
| Tool Runtime | 统一执行文件系统与命令工具 |
| Patch Engine | 负责精确修改文件 |
| Validator | 运行测试、构建、格式化，并解析结果 |
| Session Store | 保存当前会话、操作记录、结果摘要 |

## 11. 关键约束

- 只允许操作当前工作区
- 命令执行需要白名单或规则限制
- 删除、覆盖、批量移动必须二次确认
- 单次上下文读取文件数要有限制，避免失控
- 工具调用失败时必须返回清晰错误，而不是静默失败
- 所有写操作应可记录，便于回退和审计

## 12. 非功能要求

- 首次响应时间：`<= 5 秒` 给出计划或首个动作
- 单轮简单任务完成时间：`<= 30 秒`
- 工具调用失败可感知率：`100%`
- 高风险操作误执行率：`接近 0`
- 对中小型仓库稳定可用：`< 5000 文件` 的项目优先优化

## 13. MVP 验收标准

以下 5 个场景全部跑通，才算“功能闭环”成立：

1. 用户让系统新增一个小功能模块，系统能创建 2 到 5 个文件并写入代码。
2. 用户让系统修改已有逻辑，系统能先搜索定位，再补丁修改，而不是整文件覆盖。
3. 用户让系统删除或重命名文件，系统能执行并反馈结果。
4. 用户让系统运行 `test` 或 `build`，系统能读取失败输出并至少尝试修复一次。
5. 用户在上一轮基础上继续追问，系统能利用本轮上下文继续修改，不需要重新从头理解。

## 14. 建议版本节奏

### V0（1-2 周）

- 打通 CLI、LLM、文件读写、搜索、命令执行
- 支持新增文件和简单修改
- 支持结果展示

### V1 MVP（3-5 周）

- 加入稳定的 patch 修改
- 加入风险确认
- 加入测试失败后的二次修复
- 加入会话上下文保持

### V1.1（后续增强）

- Git diff
- `/undo`
- 更强的多文件重构
- 更好的命令白名单与沙箱

## 15. 成功指标

- 任务完成率
- 单轮任务平均完成时间
- 用户接受变更的比例
- 命令执行后自动修复成功率
- 用户手动接管次数
- 修改后可运行率

## 一句话产品定义

这是一个面向本地代码仓库的 CLI 代码代理，能通过自然语言完成“读代码、改代码、管文件、跑命令、看结果、继续修”的最小工程闭环。
