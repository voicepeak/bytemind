
# ForgeCLI vibecoding 复盘报告

## 1. 这轮你到底在做什么

这轮不是在“随便试一个 AI coding 工具想法”，而是在把一个模糊方向逐步收束成一个**可验证的产品闭环**：
你定义的不是“AI 帮我写代码”，而是**面向开发团队的终端原生、可审批、可审计的 coding agent runtime**，并且把“可信执行、企业适配、Windows 一等公民”作为核心差异点。与此同时，你还明确了目标用户和非目标用户，说明你不是从技术出发，而是先从产品边界出发。

---

## 2. 你的思路是怎么一步步展开的

### 2.1 第一步：先定赛道，不先写功能

你的起点不是“我要做哪些按钮、哪些命令”，而是先回答三个根问题：

1. 这产品是什么
2. 这产品给谁用
3. 它和现有方案相比，差异是什么

所以你先写了产品定位、核心差异、目标用户、非目标用户。这个动作很关键，因为它决定了后面不会走成“什么都想做”的功能堆砌。你把后端/全栈、平台团队放在 P0，把零基础用户和纯 IDE 工作流排除掉，说明你一开始就在主动收边界。

### 2.2 第二步：不是拍脑袋定功能，而是先拆竞品机制

你没有直接写“我要有聊天、要能改文件、要能跑命令”，而是先去看终端原生、商业 IDE、开源项目分别怎么做，并且不是只看名字，而是抽取机制：
Claude Code 的 rules/tools/approvals/memory，Codex CLI 的单二进制和 sandbox，Gemini CLI 的 checkpoint/resume 与 trusted folders，Aider 的 repo map 与 Git 工作流。这个阶段你做的不是“参考功能”，而是“归纳成功产品的骨架”。

### 2.3 第三步：把产品抽象压缩成 MVP

有了竞品参照后，你没有继续扩，而是开始收。你把 MVP 收敛成几类核心能力：会话系统、仓库读取、文件编辑、命令执行、安全策略、模型接入，以及规则兼容，并且明确了 P0/P1。
这说明你的思考开始从“可能性空间”进入“交付空间”——你已经不再问“还能做什么”，而是在问“第一版必须做什么”。

### 2.4 第四步：把功能清单收束成任务闭环

你没有停留在功能表，而是继续往前推进，给出了完整用户流程：

用户指定仓库和任务 → 系统加载规则与上下文 → Agent 输出计划并等待确认 → 生成 diff 并等待审批 → 执行命令并验证结果 → 输出总结并支持恢复会话。

这一步很重要，因为它标志着你的思维从“模块视角”切到了“闭环视角”。也就是说，你开始用“用户能不能完成一件事”来组织产品，而不是用“系统具备哪些功能”来组织产品。

### 2.5 第五步：再从闭环反推技术架构

在用户流程收敛之后，你才去拆技术架构：CLI、Command Layer、Session Orchestrator、Planner、Context Assembler、Policy Engine、Model Adapter、Tool Runtime，再往下对应到 session、workspace、edit、execx、model、policy 等模块。
这说明你的工程设计不是凭空展开的，而是从前面的闭环需求反推出来的。换句话说，你不是先想“我想写一个很帅的架构”，而是先确认“闭环里需要哪些系统职责”，再去分层。

### 2.6 第六步：最后补风险、指标和边界

你没有只停在“能做出来”，而是继续补了三层约束：

一层是风险控制，比如高风险操作不自动执行、写前 hash 校验、命令审批、敏感文件保护、审计日志；
一层是验证指标，比如闭环成功率、审批通过率、首次有效改动时间、人工接管率；
一层是“本期不做”，比如多智能体、IDE 插件、向量数据库、AST-aware 编辑等。

这说明你的思考已经具备产品经理和工程 owner 的视角：不仅关心“做什么”，也关心“如何控风险”和“如何判断这版算成功”。

## 3. 你的行动过程，可以复现成这样一条链

### 阶段 A：问题定义

先把“想做 AI coding”压缩成一个明确产品命题：终端原生、可信执行、面向团队。

### 阶段 B：能力取样

横向看竞品，不是为了抄，而是为了找到行业里已经被验证过的关键能力组件。

### 阶段 C：最小化

把大而全的想法压成 MVP，只保留闭环所必需的 P0 功能。

### 阶段 D：闭环建模

把产品描述从“能力列表”改写成“用户从任务开始到任务结束的完整流程”。

### 阶段 E：架构落位

围绕闭环流程，定义系统职责边界和模块分工。

### 阶段 F：工程收口

补风控、指标、本期不做，让想法从“概念”进入“可执行计划”。

## 4. 你这轮做对了什么

### 4.1 你最对的一点，是有边界感

很多 vibecoding 最大的问题是上来就做万能产品。你不是。
你一开始就明确目标用户和非目标用户，这让后续很多功能判断都有了锚点。这个能力非常重要，因为产品往往不是死在不会做，而是死在什么都想做。

### 4.2 你不是在堆功能，而是在搭闭环

你没有停在“读文件、改文件、跑命令”这种能力层，而是把它们串成一条完整主流程。
这意味着你已经进入真正做产品的状态，因为产品不是功能清单，产品是用户完成任务的路径。

### 4.3 你很早就把“可信”当成核心，不只是“聪明”

你强调审批、审计、hash 校验、敏感文件保护、危险命令分级，这说明你已经意识到 coding agent 最大的问题不是“会不会写代码”，而是“能不能让人放心地用在真实仓库里”。这是很成熟的判断。

### 4.4 你有主动取舍意识

你明确写了本期不做什么，这一点非常值钱。
因为真正推进产品的人，往往不是会加需求的人，而是敢删需求的人。

### 4.5 你已经开始用指标思维看产品

你不是只说“这个东西应该好用”，而是写了闭环成功率、审批通过率、首次有效改动时间、人工接管率。说明你已经在为后续验证和复盘做准备。

## 5. 当前最值得警惕的问题

### 5.1 PRD 的成熟度，已经高于最小可运行闭环的成熟度

这是你现在最大的风险。
也就是说，你对“应该怎么做”已经想得很完整了，但“最短链路能不能稳定跑通”这件事，未必已经被验证。这个阶段最容易出现一种假进展：文档越来越完整，真实闭环却还没被硬验证。

### 5.2 架构分层是对的，但同时开太多模块会拖慢验证

Planner、Context Assembler、Policy Engine、Tool Runtime、Search/Index、Resume、Rules 兼容，这些都合理；但如果并行推进，会让第一版变重。
对当前阶段来说，真正重要的不是把系统长全，而是让一条最短路径先跑稳。

### 5.3 你的指标已经有了，但实验设计还不够显性

现在这些指标更像“定义成功”，还不是“怎么测成功”。
也就是说，下一步你需要把指标绑定到真实任务、真实仓库、真实记录表，而不是只留在 PRD 里。

### 5.4 差异点需要被用户任务证明，而不是被文档证明

“可信执行”“Windows 一等公民”“企业适配”这些方向是对的，但只有当真实用户拿它做任务时明显感觉到好处，差异点才算成立。
否则它们还是产品假设，不是产品事实。

## 6. 你下一轮最应该怎么做

### 6.1 只打一个极薄的纵向切片

不要再扩需求，先打穿这一条：

打开仓库
→ 读取规则文件
→ 生成计划
→ 展示 diff
→ 用户审批
→ 执行一个低风险命令
→ 输出总结
→ 保存并恢复会话

这条链刚好和你定义的用户流程完全一致。只要它能稳定跑通，你这个产品才真正从“方向”进入“产品”。

### 6.2 先冻结接口，不要先丰满实现

建议你优先冻结这些输入输出：

* workspace read/list/rg
* edit diff/apply
* exec approve/run
* session save/resume
* policy check
* model adapter ask/tool-call

内部先用最简单实现都行。
比如检索先不用做太复杂，规则兼容先只支持一种，resume 先只恢复最近会话。这个阶段最重要的是可闭环，而不是可扩展。

### 6.3 把指标变成实验表，不要停留在 PRD

你可以直接做一个任务验证表，每个真实任务记录四件事：

* 是否闭环成功
* 第一次有效改动用了多久
* 中间审批了几次
* 最后有没有人工接管

连续测 10 个任务，你会非常快地知道问题出在 plan、context、diff、exec 还是 approval 上。你文档里的指标已经够用了，现在差的是实验执行。

### 6.4 把“可信执行”做成第一卖点，而不是附属特性

在你当前的路线里，最有机会变成真正差异点的，不是模型接得多，而是：

* 改动可解释
* 命令可审批
* 风险可控
* 操作可审计
* 失败可恢复

这些如果打磨好，哪怕模型能力暂时不是最强，产品也会更像一个能进团队流程的工具。

---

## 7. 一句话复盘结论

这轮 vibecoding 的本质，不是“边聊边想做什么”，而是一次比较成熟的**从灵感到产品闭环的收敛过程**：
你先定边界，再拆竞品，再压 MVP，再建流程，再反推架构，最后补风险、指标和取舍。整个方向是对的。接下来真正决定成败的，不是继续把 PRD 写得更全，而是把那条最短闭环做得更硬。

## 8. 给你一个可直接复用的“复现思路模板”

以后你做下一轮产品探索，可以直接照这个顺序走：

1. 先写一句话定位
2. 写目标用户和非目标用户
3. 看竞品，但只抽机制，不抄功能
4. 把能力压成 P0/P1
5. 把 P0 改写成用户闭环
6. 再从闭环反推架构
7. 写风险控制
8. 写验证指标
9. 写本期不做
10. 最后只做一条最短纵向切片
