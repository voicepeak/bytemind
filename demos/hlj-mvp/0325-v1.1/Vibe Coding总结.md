# ForgeCLI Vibe Coding 总结

## 1. 这轮到底在做什么

这轮不是在“继续堆 AI coding 功能”，而是在把一个方向正确但实现松散的原型，收成一个更像产品的 runtime。

起点是先重新看清项目现状：

- PRD 的产品判断是对的，尤其是“可信执行、Windows 友好、适合团队使用”这几个方向
- 代码实现明显落后于 PRD，很多关键能力只是写在文档里，还没有真正进入主流程
- 当前最大的风险不是功能少，而是“会跑”和“可被信任”之间差得还很远

所以这轮的核心，不是扩能力，而是收主链路。

一句话概括：从“能跑的 agent demo”往“可信的最小产品闭环”推进。

---

## 2. 这轮思路是怎么展开的

### 2.1 先做项目诊断，不先写代码

先通读了项目代码、PRD、上个版本总结和你的 Vibe Coding 总结，得到几个明确判断：

- Go CLI 本体很薄，`gstack-main` 更像外部 skills 资产，不是已经深度集成的内核能力
- 主循环里审批、审计、policy、session 这些“可信执行”组件存在，但没有被统一接入
- 路径安全、敏感文件保护、命令执行控制都存在可以绕过的口子
- 会话保存更像聊天记录，而不是可恢复的执行轨迹

这一步非常关键，因为它把问题从“功能点不够多”改写成了“运行时主闭环不够硬”。

### 2.2 先收口，不扩张

随后定下一个很明确的策略：

- 不先接更多 provider
- 不先深挖 `gstack-main`
- 不先做更花的界面
- 先把一条最短闭环做实

这条闭环就是：

用户输入任务 -> 模型规划 -> 用户确认 -> 调用工具 -> 审批执行 -> 审计落盘 -> 输出总结 -> 支持恢复会话

这意味着本轮工作的重心，从“功能列表”切到了“产品状态机”。

### 2.3 优先修“假的可信”

真正阻碍产品化的，不是模型聪不聪明，而是运行时有没有边界。

所以这轮优先处理的是：

- 工作区路径 containment
- 文件写入前后的 hash 校验
- 原子写入
- 命令执行审批和风险分级
- 所有关键操作统一审计
- 会话事件化

这本质上是在把“可信执行”从 PRD 文案改成代码约束。

### 2.4 再处理真实使用时暴露出来的交互问题

底座收完之后，开始暴露第二层问题：产品交互。

这些问题不是代码结构层面的，而是用户一上手就会觉得“不像产品”的地方，比如：

- 输入“你好”居然也先出一个 Plan
- `/review` 只会打印一整页 skill markdown
- 明明要求它创建文件，却返回“无法创建”
- 你拒绝一次命令审批后，它还继续问后面的命令

这一步很像真正的 vibe coding：不是先把系统做得更大，而是在真实体验里不断发现“产品感哪里断了”，然后一轮轮补上。

### 2.5 最后补上“面向用户输出”的最后一道闸

在工具链和交互基本稳定之后，又补了一层 final response review。

这层的目的不是再做一次推理，而是让最终展示给用户的话：

- 不夸大执行结果
- 不把没做的事情说成做了
- 在审批被拒绝时明确说明“没有执行”
- 文风更像正式产品，而不是工程中间态输出

这一步很重要，因为很多 agent 产品不是死在能力不够，而是死在“输出不够稳，不够像可交付物”。

## 3. 这轮具体做了什么

### 3.1 把运行时主链路收硬

本轮完成的主干改造包括：

- 重做工作区安全边界，修掉原来依赖前缀字符串判断的路径校验
- 重做编辑流程，加入 hash 校验、diff 预览、审批、原子写入
- 统一命令执行管道，支持风险分类、审批、超时和审计
- 让 policy 真正进入读写删执行主路径
- 会话从 message 记录升级为 event log
- 新增 `sessions` / `resume` 等恢复能力

这部分的结果是：ForgeCLI 终于开始具备“可信 runtime”该有的底层约束。

### 3.2 把聊天和任务执行拆开

之前所有自然语言输入都容易被当成任务，于是连“你好”都会触发计划流程。

现在改成了两种模式：

- 聊天模式：直接回复，不强制计划
- 任务模式：进入 `plan -> confirm -> tool execution`

这个改动的价值不在技术本身，而在于它把“用户是在聊天，还是在交代任务”这件事明确了。

### 3.3 把 skill 从“展示文档”改成“真正可调用”

之前 `/review` 的行为，本质上只是把 `SKILL.md` 打出来。

现在改成：

- `/review`：激活 skill
- `/review <任务>`：带着 skill 直接执行任务
- `/clear-skill`：清空当前 skill

这让 skill 不再只是资料，而是进入运行时上下文的一部分。

### 3.4 收紧 tool use 行为

用户已经明确让它创建或修改文件时，系统不能允许模型只靠自然语言回复“无法创建”。

因此这轮又补了两件事：

- 对创建/修改/删除类任务，提示词更强约束工具调用
- 如果模型第一轮仍然没有正确调工具，CLI 会再追问一次，要求它使用相应工具

这一步的目标很明确：把“会说”收敛成“会做”。

### 3.5 修掉拒绝审批后的反复追问

这是后来真实使用里暴露出的典型产品问题。

问题表现是：你已经对 `execute_command` 输入了 `n`，但系统还继续弹新的命令审批。

原因是旧逻辑把“用户拒绝执行”当成普通 tool result，又喂回了模型；模型就会继续尝试下一条命令。

现在改成了：

- 只要用户拒绝任一带审批的变更类工具
- 当前任务立即结束
- 不再继续追问新的命令或写入动作

这个修复非常像产品化阶段的真实工作：问题不大，但不修掉，体验就会显得很蠢。

### 3.6 给最终输出加一层 AI 审查

最终回复现在会多走一层 reviewer：

- 检查表述是否规范
- 检查有没有虚报执行结果
- 检查审批被拒绝时是否说明清楚
- 在不改事实的前提下把输出收得更专业

这件事直接提升了交付感。

## 4. 这轮最有价值的几个判断

### 4.1 先把“可信”做实，价值大于继续扩模型能力

这轮最正确的决策，就是没有继续把注意力放在“支持更多模型、更多技能、更多花哨能力”上，而是先把运行时边界收紧。

因为对 coding agent 来说，真正稀缺的不是“会不会写”，而是“能不能放心让它碰真实仓库”。

### 4.2 文档成熟度不能长期领先实现成熟度

这轮一开始最明显的风险，就是 PRD 已经写得像产品成型了，但代码主链路还没有真到那个程度。

如果不及时把文档里的承诺变成代码里的约束，项目很容易陷入一种危险状态：

- 文档越来越漂亮
- 产品越来越像“已经完成”
- 但真实可交付性并没有同步提升

这轮的收口工作，本质上是在把这个偏差拉回来。

### 4.3 产品感来自“异常路径”是否稳定

不是只有“正常完成任务”才体现产品质量。

这轮暴露出来的几个最典型问题，恰恰都在异常路径上：

- 闲聊被误识别为任务
- skill 只展示不执行
- 工具未调用导致“口头完成”
- 审批被拒绝后还继续追问

这些问题都不是大功能，但每一个都直接决定用户会不会觉得“这东西像产品”。

### 4.4 真正的 vibe coding，不是乱写，而是快速收敛

这轮其实很能说明一个事实：

高质量的 vibe coding，不是想到什么就加什么，而是基于真实体验快速发现问题、缩短反馈回路、持续把系统收得更稳。

换句话说，这轮的 vibe coding 不是“发散”，而是“收敛”。

---

## 5. 当前版本处在什么阶段

如果给这轮结束时的 ForgeCLI 一个定位，我会这样定义：

它已经不是单纯的 demo 了，但也还不是完整成熟产品；
它现在更接近一个“可信最短闭环已经打通的 runtime MVP”。

已经具备的产品气质：

- 有运行时边界
- 有审批
- 有审计
- 有事件化会话
- 有 skill 激活
- 有最终输出审查

还没完全到位的地方：

- `gstack-main` 仍然主要是外部 skills 资产，不是深度内核集成
- repo context 仍然偏基础
- provider 抽象还不够完整
- 更系统的集成测试和更完整的交互设计还需要继续补

---

## 6. 这轮给你的方法论启发

如果把这轮再抽象一层，它给出的不是一个单独项目的经验，而是一套很适合继续复用的工作方法：

1. 先诊断现状，不急着写代码
2. 先收主闭环，不急着扩能力
3. 先修安全和可信，再修“聪明程度”
4. 用真实交互把问题逼出来
5. 优先修那些直接破坏产品感的问题
6. 最后给输出再加一道质量闸门

这套节奏非常适合 agent 类产品，因为 agent 产品天然容易出现“看起来很强，实际很飘”的问题。

---

## 7. 一句话总结

这轮 Vibe Coding 的本质，不是继续发散产品想象力，而是把已经判断正确的方向，硬生生收成一版更像产品的运行时。

真正的进步不在于“会更多工具了”，而在于 ForgeCLI 开始有了边界、状态机、异常处理和交付感。

---

## 8. 后续几轮补充收口

前面的总结更偏“从运行时底座到可信闭环”的收口，但这轮后半段其实又继续往前推了一步：开始处理那些真正影响用户观感的终端交互问题。

### 8.1 从“把过程露给用户”转向“只把结果交给用户”

在真实使用里，后面又暴露出一个很典型的问题：

- 模型在后台读文件、按行读取、搜索测试文件、grep 内容
- 这些步骤本来只是为了得到结论
- 但 CLI 把这些内部 tool trace 直接打印给了用户

这会让产品看起来像调试台，而不是面向用户的成品。

所以后续又做了一层非常关键的收口：

- 只读型后台工具默认不直接展示
- 前台只显示一次“思考中...”
- 真正需要用户介入的审批类操作才继续显示
- 用户最终看到的是结论，而不是中间检索流水账

这一步意义很大，因为它标志着 ForgeCLI 不再只是“让模型能调用工具”，而是开始认真区分：

- 哪些信息是给模型的
- 哪些信息是给用户的

### 8.2 异常路径继续暴露出“工具描述”和“工具实现”之间的缝

后面还有一类问题也很有代表性：

- `glob` 连续空结果
- `list_files` 在根目录场景下返回空
- 模型因此反复试探不同只读工具，最后撞到最大迭代次数

这说明一个很现实的问题：
agent 产品里，光有工具还不够，**工具对模型宣称的能力必须和真实实现一致**。

这一轮补的不是新功能，而是把这些隐性错位补平：

- 修 `glob` 对常见模式的匹配
- 修根目录 `list_files` 的过滤逻辑
- 连续空结果时强制收尾，不让任务无限空转

这本质上又是一次典型的 vibe coding 收敛：
不是“这个模型不行”，而是“运行时对失败路径缺少约束”，所以要补的是产品状态机，而不是抱怨模型表现。

### 8.3 交互细节开始进入“产品感”层面

随着问题一轮轮被逼出来，这轮后面已经不只是修逻辑，也开始修“终端里的产品感”：

- 激活 skill 之后，在输入提示里直接显示当前技能名
- 同时给出退出技能的入口
- 启动 banner 直接告诉用户如何查看技能和工具命令
- 把用户可见文案统一收成中文，不再中英混杂

这些改动不大，但非常重要。

因为它们意味着这个项目开始从“工程上能工作”进一步转向“用户拿起来知道怎么用、知道自己在哪个状态里”。

### 8.4 对 `gstack-main` 的判断也更清楚了

到这一步，`gstack-main` 的角色其实已经非常明确：

- 它仍然是 ForgeCLI 当前 skill 体系的资源来源
- 不是运行时内核的一部分
- 但在没有迁移到你自己的轻量 skills 目录之前，也不能直接删

这说明项目现在的技术边界也更清楚了：

- ForgeCLI 本体负责 runtime
- `gstack-main` 负责外部 skill 资产

这比一开始那种“看起来什么都在一起”的状态要健康得多。

---

## 9. 这轮额外带来的方法论补充

如果把后半段再抽象一层，其实又补出了几条很有价值的经验：

1. 不要把 agent 的内部检索过程直接暴露给用户
2. 终端产品也需要明显的“状态提示”和“退出路径”
3. 工具描述和工具实现必须对齐，否则模型会持续空转
4. 很多所谓“模型问题”，本质上是运行时缺少失败路径治理
5. 真正的产品化，往往发生在这些不起眼的交互细节上

---

## 10. 补充一句话结论

如果说前半段是在把 ForgeCLI 做成“可信的 runtime MVP”，那么后半段做的事，就是开始把它收成“用户真的愿意持续使用的终端产品”。

## 8. 后续补充：从“能解释”继续收成“像产品”

前面的总结更多是在讲这轮主干改造；但在真正把二进制交给你持续试用之后，又暴露出一批更细、也更像真实产品打磨阶段的问题。

这些问题有一个共同点：

- 它们不一定会让系统彻底不能用
- 但会立刻破坏“这是个成品”的感觉

这批问题主要包括：

- review 场景会把代码片段、grep 结果甚至二进制脏内容刷到终端
- 只读审查任务里，模型会反复试探工具，出现空转
- 计划和最终回复里偶尔还会夹带大段代码
- skill 虽然已经能激活，但用户在界面上感知不到“当前正处于哪个 skill”
- 前台看到了大量后台检索步骤，而不是“正在思考 -> 给出结果”

这意味着项目进入了一个新的阶段：
不是继续修主链路有没有通，而是开始修“前台应该让用户看到什么，不该看到什么”。

### 8.1 你开始从“agent 调试视角”切换到“客户交付视角”

这是这轮后半段最有价值的变化之一。

你不再只是问：

- tool 有没有调用成功
- skill 有没有加载
- 计划和执行链路有没有走通

而是开始问：

- 为什么用户会看到这么多内部步骤
- 为什么后台读取、搜索、未命中提示会直接暴露出来
- 为什么当前处于哪个 skill 不够明确
- 为什么界面里还留着英文和工程味

这其实说明你已经从“做一个能工作的 agent”进入了“做一个能给别人用的产品”。

### 8.2 这轮后续迭代，本质上是在收前台信息边界

前面那轮主要收的是：

- 安全边界
- 执行边界
- 审批边界
- 会话边界

而这轮后续补丁收的是另一种边界：

- 什么是内部过程
- 什么是对用户可见的信息
- 什么应该静默在后台
- 什么必须显式展示给用户

于是后续又补上了几件很关键的事：

- 读文件、搜文件、grep、项目结构这类只读工具默认不再直接打印
- 前台改成只显示一次“思考中...”，然后给出最终结果
- 真正需要用户参与的审批类步骤仍然保留可见
- 计划输出和最终回复增加硬清洗，防止模型贴大段代码
- 对连续空结果和工具空转增加强制收尾

这组改动的价值，不在于模型能力变强了，而在于产品开始学会“克制”。

### 8.3 skill 的产品形态开始变清楚了

skill 早期只是被成功“激活”；但真正产品化时，激活不够，用户还需要持续感知当前状态。

所以后面又继续补了：

- 输入提示符里直接显示当前 skill 名称
- 同时给出退出 skill 的命令
- 启动 banner 直接展示“查看技能”和“查看工具/命令”的入口
- 技能帮助文案、状态文案和初始界面继续统一成中文

这一步很小，但很关键。因为一旦 skill 有了“可见状态”和“可退出状态”，它才更像一个真实的模式，而不是一段隐式 prompt。

### 8.4 真实试用继续暴露了运行时和提示之间的不一致

后续测试里还暴露出另一类典型问题：

- `glob` 对模型常用模式支持不稳定
- `list_files(dir='.')` 这种根目录查询会被错误过滤为空
- 模型在“看看 calculator.py”这类输入下，有时会掉回聊天路径

这类问题其实比明显报错更烦，因为它们不会总是炸，而是会制造“不稳定、不像成品”的感觉。

所以后来又继续补了：

- `glob` 模式支持收紧
- 根目录文件过滤修正
- 文件路径感知增强，让这类输入稳定进入任务态
- 空转时强制总结而不是无限读下去

这说明真实使用永远比纸面设计更诚实。

### 8.5 对 `gstack-main` 的判断也进一步清晰了

在后续对话里，你还专门问了 `gstack-main` 能不能删。

这个问题很有代表性，因为它把之前那个判断又进一步坐实了：

- `gstack-main` 现在本质上仍然是外部 skills 资产目录
- ForgeCLI 主体并不依赖它来完成读写执行本身
- 但当前 skill 体系仍然直接从这个目录加载

也就是说，它不是内核的一部分，但当前又确实承担着 skill 资源仓的角色。

这让项目结构的现实情况更明确了：
ForgeCLI 核心 runtime 已经越来越像自己的产品；
但 skill 生态层仍然带着一块外部资产包的影子。

### 8.6 这轮给你的另一个方法论启发：不要只修“能不能用”，还要修“用户看见什么”

如果说前半段的经验是：

1. 先诊断
2. 先收主链路
3. 先修可信边界

那后半段的经验可以补成：

4. 把真实用户看到的每一段输出都当成产品界面
5. 后台过程默认隐藏，必要状态才暴露
6. 让模式可见，让退出路径可见，让入口可见
7. 发现空转和噪音时，不优化解释，直接优化显示边界

这套补充方法其实很重要，因为 agent 类产品很容易“后端很强，前台很吵”。

### 8.7 这轮结束时的更准确定位

如果说前面总结里把它定义为“可信最短闭环已经打通的 runtime MVP”，那么在这轮后续修补之后，可以再多加一句：

它开始从“工程上成立”进一步走向“交互上也开始像成品”。

这不代表项目已经成熟了；而是意味着你已经从“把能力做出来”进入了“把能力收成体验”的阶段。
