# ForgeCLI 当前版本总结

## 版本定位

这版的核心目标不是继续扩功能，而是把 ForgeCLI 从“能跑的 agent 原型”收成“更可信、更像产品的 runtime MVP”。

本轮最重要的变化，不是新增了多少能力，而是把原来分散、松散、容易绕过的执行链路收成了一条真正可控的主流程。

---

## 这版已经完成了什么

### 1. 可信执行底座已经收起来了

当前版本已经具备这些基础能力：

- 工作区路径 containment 校验，避免越界读写和相邻目录误判
- 统一的变更执行管道：路径解析 -> policy 检查 -> diff/命令预览 -> 用户审批 -> 执行 -> 审计 -> 会话落盘
- 文件编辑改为写前 hash 校验 + 临时文件写入 + rename 原子替换
- `execute_command` 进入统一审批和风险分级
- 审计日志写入真实 `session_id`
- 会话支持 event log，并支持 `sessions` / `resume [id|latest]`

这意味着“可信执行”已经不再只是文档概念，而是进入了真实主路径。

### 2. CLI 交互已经更像产品，而不是工程实验

这版又补了几项直接影响体验的改动：

- 聊天模式和任务模式拆开，输入“你好”不再错误触发计划流程
- skill 从“打印整篇 markdown”改成“真正激活并参与执行”
- 支持 `/review`
- 支持 `/review <任务>`
- 支持 `/clear-skill`

这部分的价值在于：用户开始能把 ForgeCLI 当成一个有状态的助手，而不是只会吐中间过程的调试器。

### 3. tool use 的稳定性明显增强

围绕“创建文件、修改文件、执行命令”这类任务，系统现在更强地要求模型真正调用工具，而不是只靠自然语言声称完成。

补上的关键行为包括：

- 对创建/修改/删除类任务，执行提示词更明确要求走工具
- 模型第一轮没有正确调工具时，CLI 会追加一次强制性纠偏
- 用户拒绝执行任一带审批的变更类工具后，当前任务立即结束，不再继续弹后续命令

这几项改动一起解决了前面真实使用中暴露出的两个典型问题：

- 明明要求创建 Python 程序，却返回“无法创建”
- 已经拒绝命令审批了，系统还继续反复追问

### 4. 最终回复现在会再经过一次 AI 审查

在真正展示给用户之前，最终回复现在会再经过一轮 reviewer：

- 检查措辞是否规范
- 检查有没有虚报执行结果
- 审批被拒绝时明确说明没有执行
- 在不改事实的前提下把输出收得更专业

这一层不会改变底层 tool 执行，只作用于最终展示文本，但对产品观感帮助很大。

---

## 这版重点修掉了哪些问题

### 1. 假的安全边界

之前路径安全主要依赖前缀字符串判断，存在工作区边界误判风险。现在已经收成更严格的 containment 检查。

### 2. 审批、审计、policy 没真正进入主流程

之前很多安全组件只存在于包里，没有真正控制主链路。现在写、删、建目录、执行命令都走统一审批和审计。

### 3. 会话只像聊天记录

之前 session 更像 message 保存；现在已经能记录关键 event，开始接近可恢复任务会话。

### 4. 闲聊也触发任务计划

现在聊天和任务分流，避免“你好”也先出一个 `Plan`。

### 5. skill 只能展示，不能真正调用

现在 `/review` 不再刷整页 `SKILL.md`，而是作为活跃 skill 进入执行上下文。

### 6. 用户拒绝一次命令后还会继续追问

现在只要审批被拒绝，当前任务立即停止，不再继续尝试后续命令。

---

## 当前版本仍然不是终点

这版已经明显比之前更像产品，但它依然是 runtime MVP，不是完整成熟版。

还没有完全做完的主要包括：

- 更强的 repo context 构建
- 更系统的集成测试
- 更完整的 provider 抽象
- 把 `gstack-main` 的能力从“skills 资产”进一步内核化
- 更细的 CLI 展示和状态反馈

---

## 一句话结论

这版最重要的变化，不是 ForgeCLI 会了更多工具，而是它开始具备一个产品最基本的样子：

有边界，有审批，有审计，有会话，有 skill，有异常路径控制，也有面向最终用户的输出质量把关。

## 补充更新：这版后续又完善了什么

在上面的基础上，这个版本后面又继续做了几轮非常具体的交互收口，重点不再是“会不会调用工具”，而是“用户到底会看到什么”。

### 1. 后台只读工具默认不再直接暴露

现在像这些内部只读步骤：

- `read_file`
- `read_file_lines`
- `list_files`
- `glob`
- `grep`
- `get_project_structure`

默认不会再把 tool trace 直接打印给用户。CLI 只会在第一次后台检索时显示一次“思考中...”，然后直接给最终结论。

真正还会显式展示的，是需要用户介入的变更类工具，比如：

- `write_file`
- `edit_file`
- `create_directory`
- `delete_file`
- `execute_command`

这让 ForgeCLI 从“把内部过程全摊开”变成了“前台给结果，后台跑检索”。

### 2. tool 空转问题又收了一轮

后续真实使用中还发现了新的问题：

- `glob` 对模型常用模式的支持不够稳定
- `list_files` 在根目录过滤场景下会误返回空
- 连续空结果后模型会继续试别的只读工具，最后撞到最大迭代次数

针对这几个问题，这版后续又补了：

- `glob` 模式匹配增强
- 根目录 `list_files` 过滤修正
- 连续空结果或工具调用过多时，强制停止继续探测并直接总结

这部分让“读型任务”明显更稳了。

### 3. 技能态开始有真正的状态感

后续又增强了 skill 的终端体验：

- 激活 skill 后，输入提示会显示当前技能名
- 同时明确显示退出技能的入口 `/clear-skill`
- `status` 里也会显示当前技能和退出提示

这意味着 skill 不再只是“隐藏上下文”，而是进入了用户可感知的运行状态。

### 4. 启动界面和帮助文案继续产品化

启动 banner 现在会直接提示：

- `skills`：查看技能
- `help`：查看工具与命令
- `/clear-skill`：退出当前技能

同时，用户可见的界面文案进一步统一成中文，减少中英混杂带来的工程感。

### 5. 关于 `gstack-main` 的当前判断

这版后续也重新确认了 `gstack-main` 的角色：

- 现在它仍然是当前 skill 体系的资源目录
- ForgeCLI 主体运行并不依赖它的浏览器/QA/runtime 内核
- 但在没有迁移到新的轻量 skills 目录之前，不能直接删掉，否则 `/review` 等 skill 会失效

也就是说，当前项目里：

- ForgeCLI 是 runtime
- `gstack-main` 是外部 skill 资产包

这个边界现在已经比之前清楚得多。

## 补充更新：后续交互收口

在上面这些主干改造之后，这个版本后续又继续补了一轮“前台体验层”的修正。重点不是新增能力，而是继续减少噪音，让 CLI 更像产品。

### 1. 后台读工具默认隐藏

现在读文件、读指定行、列文件、glob、grep、项目结构、依赖分析这些只读工具默认不再直接打印到前台。

前台行为改成：

- 后台开始检索时，只显示一次 `思考中...`
- 检索过程默认静默
- 最后直接给出结果

真正还会继续展示的，主要是需要用户介入的审批类步骤，比如写文件、删文件、创建目录、执行命令。

这解决了之前“内部 tool 过程刷屏”“未命中结果像报错一样堆在终端里”的问题。

### 2. 计划和最终回复继续加硬清洗

即使中间 tool trace 已经收住，如果模型在计划或最终回答里继续贴大段代码，体验仍然会很差。

所以这版后续又补上了：

- 计划输出的代码块清洗
- 最终回复的代码块和 plain code block 清洗
- 只有用户明确要求“贴完整代码”时，才允许原样展示源码

这让输出更稳定，也更符合“默认给结论，不默认给原文”的产品方向。

### 3. review 场景继续收紧

这版后续继续强化了只读审查场景：

- review 默认只读
- review 中的后台搜索和读取不再直接展示
- 遇到连续空结果和工具空转时，会提前强制总结，而不是一直试探到最大迭代次数

同时还修了几处底层不稳定点：

- `glob` 对常见模式支持更稳
- `list_files(dir='.')` 不再错误返回空
- 二进制和构建产物不再被误扫进 grep 结果里

### 4. skill 的界面感更明确了

后续又继续补了 skill 的前台状态展示：

- 激活 skill 后，输入提示符会直接显示当前 skill 名称
- 同时显示退出 skill 的命令 `/clear-skill`
- 初始界面直接展示“查看技能”和“查看工具与命令”的入口
- `status` 中也会明确显示当前技能和退出方式

这让 skill 不再只是后台 prompt 状态，而更像一个真正可见、可退出的模式。

### 5. CLI 前台文案继续中文化

这一轮后续还把前台能看到的文案继续统一成中文，例如：

- `Skills: N loaded` 改成 `已加载技能：N 个`
- `Plan:` 改成 `计划：`
- `current skill` 改成 `当前技能`

整体目标很明确：减少工程味、减少英语残留、减少内部术语直接暴露。

---

## 当前版本的更准确定义

如果说前面的总结把这版定义成“可信 runtime MVP”，那么补上后续这些交互修正之后，可以更准确地说：

这版已经不只是“能力上能跑通”，而是开始在“用户看到什么、看不到什么、什么时候等待、什么时候确认”这些产品细节上进入收口阶段。

也就是说，这版的价值不只在底层可信，还在于前台开始学会克制。
