# ByteMind PRD

## 0. 文档信息

- 产品名称：ByteMind
- 文档版本：PRD 1.0
- 产品形态：本地 AI Coding TUI
- 目标平台：macOS / Linux / Windows
- 技术实现：Go

## 1. 产品愿景与定位

### 1.1 一句话定义

ByteMind 是一个运行在本地代码仓库中的 AI Coding TUI，帮助开发者在终端内完成“理解项目、修改文件、执行命令、查看结果、恢复会话、继续迭代”的最小工程闭环。

### 1.2 产品定义

ByteMind 是一款面向本地代码仓库的 AI Coding TUI，帮助开发者在一个可持续会话的终端界面中完成“理解项目、修改代码、执行命令、验证结果、恢复会话、继续任务”的工程闭环。

ByteMind 以 `TUI` 作为主交互形态，以 Slash 命令和单次 `run` 作为补充入口。产品目标不是替代 IDE，也不是做成沉重的平台，而是在开发者最常用的 Terminal 场景里，提供一套足够轻、足够稳、足够可控的工程执行助手。

> ByteMind 是一个运行在本地工作区中的 AI Coding TUI，让开发者在 Terminal 内完成“理解 -> 修改 -> 执行 -> 验证 -> 恢复”的最小闭环。

### 1.3 产品定位与形态选择

ByteMind 选择 `TUI` 作为主形态，原因如下：

1. 更符合当前产品形态  
相比纯 CLI 对话，TUI 更适合承载状态栏、会话列表、帮助面板、审批弹窗和命令面板，让“执行中的产品”比“命令行里的聊天”更清晰。

2. 更符合目标用户习惯  
核心用户本身就在 Terminal 内工作，希望少切换、少跳出、少重复解释上下文。

3. 更贴合当前代码基线  
当前仓库已经具备启动页、会话页、会话恢复、审批弹窗等 TUI 能力，产品主形态应与真实实现保持一致。

## 2. 用户痛点与需求分析

### 2.1 立项背景

开发者对 AI 编程工具的需求，正在从“回答问题”转向“直接执行任务”。仅提供问答、解释和补全，已经很难满足真实开发流程中的连续工作需求。

### 2.2 用户痛点

三类用户的共性痛点可归纳为：

- 项目理解成本高，AI 需要反复补上下文
- 很多 AI 工具只能建议，不能真实落盘和执行
- 修改后的结果不可追踪，用户看不清实际变更
- 测试、构建和修复过程不连贯，闭环常被打断
- 中断后难恢复，任务容易重新分析一遍
- 用户既希望 Agent 能执行，又担心它乱执行

### 2.3 需求归纳

用户真正需要的是：

- 能直接读取仓库上下文，而不是只根据几段粘贴代码回答
- 能修改文件、运行命令、检查结果，而不是只给建议
- 能保持任务连续性，在多轮交互中持续推进同一工程问题
- 能在具备执行能力的同时，保持边界清晰、过程透明、风险可控

## 3. 产品目标

### 3.1 产品目标

1. 让用户能够在本地工作区中通过自然语言完成真实的软件工程操作。
2. 让用户在 TUI 会话中持续推进一个任务，而不是只能进行一次性问答。
3. 让系统具备项目理解、代码修改、命令执行和结果验证能力，形成完整闭环。
4. 让执行过程足够清晰、可恢复、可审查，提升用户对 agent 的信任。
5. 在 Go 技术栈下保持可持续迭代，严格控制范围，避免产品形态膨胀。

### 3.2 产品原则

1. 本地优先：默认围绕当前工作区运行，不依赖云端项目索引。
2. TUI 优先：TUI 是主交互入口，帮助、命令、会话和审批都应围绕 TUI 主路径展开。
3. 执行优先：重点不是“聊得像助手”，而是“做得像工具”。
4. 可控优先：文件修改、命令执行、越界操作必须具备最小可用的边界与确认机制。
5. 恢复优先：会话中断、任务打断、预算耗尽时，系统应尽量保留可继续状态。

### 3.3 成功标准

当用户输入以下类型任务时，ByteMind 应能稳定完成主要流程：

- “分析这个仓库的结构并告诉我入口在哪里。”
- “修改当前模块的一段逻辑并直接落盘。”
- “运行测试并根据结果继续修复。”
- “恢复之前的会话并继续完成剩余工作。”

闭环成立的标准是：

- 系统能读取本地上下文
- 系统能调用工具执行实际操作
- 系统能在 TUI 中给出明确反馈
- 用户知道当前做到哪一步，以及下一步如何继续

## 4. 用户画像

### 4.1 目标用户

ByteMind 的目标用户主要包括三类：

- 终端优先的后端开发者
- 多语言、跨文件工作的全栈/工具型开发者
- 关注风险边界与恢复能力的小团队技术负责人或核心维护者

### 4.2 用户画像

#### 4.2.1 小李：终端优先的后端开发工程师

基本信息：

- 29 岁，后端开发，主要负责 Go 服务和内部 API
- 日常在 Terminal、SSH、VS Code 之间切换
- 熟悉 Git、测试命令、构建命令，希望尽量在终端里闭环完成工作

工作习惯：

- 进入一个仓库后，先看目录结构、配置入口和服务启动方式
- 修改代码前通常会先全局搜索关键词，再读关键文件
- 修改后会立刻跑测试、构建或脚本验证结果
- 遇到中断时，希望第二天直接接着昨天的进度继续做

核心目标：

- 快速理解陌生仓库和陌生模块
- 让 Agent 帮他做真实改动，而不是只给建议
- 在终端里完成“分析 -> 修改 -> 验证 -> 总结”

主要痛点：

- 通用聊天式 AI 需要反复喂上下文，理解仓库成本高
- 改动以后看不清具体改了哪些文件、跑了哪些验证
- 复杂任务被打断后，恢复成本高

#### 4.2.2 小王：多语言的全栈/工具型开发者

基本信息：

- 27 岁，全栈开发，既写前端，也写后端、脚本和配置
- 经常需要在一个仓库中同时修改多个文件，或临时新增脚本与模板
- 希望工具既能在 TUI 中持续推进，也能保持结果清晰、可复制

工作习惯：

- 接到任务后，会先让工具帮他确认现有项目结构和规范
- 对重复性工作敏感，倾向于让 Agent 直接生成可落地结果
- 处理复杂任务时，希望会话里有清晰的阶段反馈，帮助自己和 Agent 对齐进度
- 关注结果是否适合整理成 issue、PR 描述或交接说明

核心目标：

- 在现有项目结构下快速新增文件、脚本、配置或模板代码
- 让一次任务尽量少切换上下文，最好在一次会话里推进完成
- 让终端 Agent 真正融入自己的开发工作流

主要痛点：

- 很多工具只能“说怎么做”，不能“替他做完”
- 多文件任务容易丢上下文，结果不稳定
- 交互路径不清晰时，很难判断任务现在推进到了哪一步

#### 4.2.3 小黄：小团队技术负责人兼核心维护者

基本信息：

- 33 岁，技术负责人，既做关键代码评审，也会亲自处理复杂改造
- 关注研发效率，也关注风险、治理和团队采纳成本
- 希望先在低风险、高频任务上引入 Coding Agent

工作习惯：

- 评估新工具时，会先看边界是否清晰、输出是否可解释、失败是否可恢复
- 希望团队成员在同一套规范下使用工具，而不是各自摸索
- 特别关心危险命令、越界改动和不可追踪的文件修改
- 对“继续上次任务”能力敏感，因为很多任务本身就是多天推进

核心目标：

- 让团队在本地仓库内安全地使用 Agent
- 降低重复分析、重复修复和重复验证的时间成本
- 让每次执行都有记录、有摘要、可回溯

主要痛点：

- 很多 Agent 看起来能力很强，但实际难治理、难解释、难落地
- 缺少清晰的审批、阻断和变更追踪机制
- 团队任务一旦中断，就很难恢复到准确的上下文状态

## 5. 典型使用场景与用户故事

### 5.1 用户故事

#### 5.1.1 小李的用户故事

- 作为小李，我进入一个陌生 Go 仓库时，希望先用 ByteMind 快速分析项目结构、配置入口、关键模块和调用链，这样我不用手动粘贴上下文就能开始工作。
- 作为小李，当我确认修改方向后，希望 ByteMind 能直接在仓库里改代码、补参数校验、修函数逻辑，并在改完后立刻运行测试或构建，这样我可以在终端里完成完整闭环。
- 作为小李，当任务做到一半被打断时，我希望第二天重新进入仓库后，ByteMind 能告诉我上次改了哪些文件、跑了哪些命令、卡在什么地方，并允许我继续执行，这样我不用从头再来。

#### 5.1.2 小王的用户故事

- 作为小王，当我需要新增脚本、配置、模板文件或一个小模块时，我希望 ByteMind 能先理解当前项目结构和约定，再直接生成并落盘结果，而不是只返回一段建议代码。
- 作为小王，当我处理一个涉及多个文件的任务时，我希望 ByteMind 能在一次会话中持续推进“先查找、再修改、再验证、最后总结”的过程，这样我能减少上下文切换和重复描述。
- 作为小王，当任务较复杂时，我希望会话里始终有清晰的阶段反馈，让我和 Agent 始终知道当前步骤、已完成内容和下一步动作。

#### 5.1.3 小黄的用户故事

- 作为小黄，当我评估 ByteMind 是否适合团队落地时，我希望它默认限制在当前工作区内操作，Shell 命令具备审批和阻断机制，高风险动作必须确认，这样我才敢让团队真实使用。
- 作为小黄，当团队成员使用 ByteMind 修改代码时，我希望它能明确告诉我改了哪些文件、执行了哪些命令、验证是否通过，这样我可以快速 review 结果，而不是面对黑盒输出。
- 作为小黄，当一个复杂任务需要跨多天推进时，我希望 ByteMind 能恢复上次会话和任务进度，并给出继续建议，这样团队不会因为上下文丢失而反复做同样的分析工作。

### 5.2 用户旅程

#### 阶段一：进入仓库

- 用户进入目标仓库目录
- 启动 `go run ./cmd/bytemind` 或 `go run ./cmd/bytemind chat`
- 进入 TUI 启动页，并确认当前工作区已就绪

#### 阶段二：理解任务与仓库

- 用户输入自然语言任务
- 系统浏览目录、读取文件、搜索关键词
- 用户确认系统已经理解项目结构和目标范围

#### 阶段三：确认方案与推进任务

- 用户与 Agent 对齐当前目标和执行顺序
- 用户通过会话持续推进当前任务
- 系统在执行过程中更新当前阶段与状态

#### 阶段四：执行修改与验证

- 系统进行代码生成或修改
- 系统运行测试、构建或脚本命令
- 若涉及高风险 Shell，系统弹出审批确认

#### 阶段五：总结与恢复

- 系统输出变更摘要、命令摘要和结果总结
- 会话被自动保存
- 若任务中断，用户下次进入时可恢复会话并继续

### 5.3 核心场景

- 仓库理解：分析项目结构、定位入口、理解调用链
- 定向代码修改：增加配置项、修复逻辑、补充校验
- 新建文件或模块：生成脚本、模板、配置和小模块
- 命令执行与验证：运行测试、构建、脚本并基于结果继续推进
- 连续迭代：在一次 TUI 会话中完成分析、修改、验证、修复
- 恢复未完成任务：继续上次会话主题与进度

## 7. 功能架构

### 7.1 系统能力分层

ByteMind 当前可分为四层核心能力：

1. 交互层  
以 TUI 为主入口，承载会话输入、状态反馈、帮助、命令面板、会话列表和审批弹窗。

2. 会话层  
负责会话创建、持久化、恢复、最近会话展示和工作区一致性校验。

3. 执行层  
负责目录浏览、文件读取、文本搜索、文件修改、补丁应用和 Shell 执行。

4. 配置与模型层  
负责 provider 接入、模型配置、流式开关、审批策略和执行预算控制。

### 7.2 工具能力与执行边界

当前内置工具包括：

```text
list_files
read_file
search_text
write_file
replace_in_file
apply_patch
run_shell
```

### 7.3 配置体系

当前核心配置项：

- `provider.type`
- `provider.base_url`
- `provider.model`
- `provider.api_key`
- `provider.api_key_env`
- `provider.anthropic_version`
- `approval_policy`
- `max_iterations`
- `stream`

按当前实现，ByteMind 会从以下位置查找配置：

- `.bytemind/config.json`
- `~/.bytemind/config.json`

并支持通过环境变量覆盖关键字段。

当前支持的 Provider：

- `openai-compatible`
- `openai`
- `anthropic`

### 7.4 研发实现映射

| 模块 | 职责 |

| `cmd/bytemind` | 启动入口、参数解析、模式路由 |

| `internal/agent` | 对话循环、工具调用、预算控制、停止总结 |
| `internal/session` | 会话创建、保存、恢复、列表 |
| `internal/tools` | 文件工具、Shell 工具 |

| `internal/config` | 配置加载、环境变量覆盖、参数规范化 |

当前实现原则：

- 单二进制优先
- 本地状态优先
- 会话文件持久化优先
- 工具能力与界面能力分层
- 跨平台终端可运行优先

## 9. MVP 定义

### 9.1 当前 MVP 形态

ByteMind 以 `TUI` 为入口：

- 无参数启动默认进入 `TUI`
- `run` 用于单次任务执行

当前 TUI 采用单面板聊天布局，不做复杂侧栏或多列信息架构。

核心界面由五部分组成：

1. 启动页  
居中 Logo + 输入框，用户按 Enter 后进入正式会话页。

2. 顶部状态栏  
展示 model 连接状态。

3. 主会话区  
展示用户消息、助手回复以及工具执行结果，保证主路径足够聚焦。

4. 底部输入区  
承载多轮对话输入与快捷提示。

5. 弹窗层  
用于帮助、命令面板、会话列表、审批确认。

`run` 模式作为补充入口，服务于：

- 单次快速任务
- 命令行脚本调用
- 不需要进入完整 TUI 会话的场景

### 9.2 当前已支持的 Slash 命令

```bash
/help
/session
/sessions [limit]
/resume <id>
/new
/quit
```

### 9.3 当前已支持的快捷操作

- `Enter`：发送输入
- `Ctrl+N`：新建会话
- `Ctrl+L`：打开最近会话列表
- `Ctrl+Up / Ctrl+Down`：按行滚动主会话区
- `PgUp / PgDown`：按页滚动主会话区
- `Home / End`：跳转到顶部或底部
- `Ctrl+C`：退出
- `Y / Enter`：批准 Shell 审批
- `N / Esc`：拒绝 Shell 审批

### 9.4 当前弹窗能力

- 帮助弹窗
- 命令面板
- 最近会话列表
- Shell 审批弹窗

### 9.5 MVP 指标

核心指标：

- 首条任务发送成功率
- 含工具调用任务的完成率
- 会话恢复成功率
- Shell 审批通过后的任务完成率
- 预算耗尽后继续任务的成功率

体验指标：

- 用户能否在一次 TUI 会话里完成“分析 -> 修改 -> 验证”
- 用户能否在不离开终端的前提下恢复并继续任务
- 用户是否能清楚感知当前状态、风险和下一步动作

## 10. 风险与挑战

### 10.1 非功能需求

可用性：

- 启动后应尽快进入可交互状态
- 主路径必须简单：启动、输入、执行、恢复
- 错误信息需可读、可继续处理

稳定性：

- 工具执行失败不能直接破坏整个会话
- 会话存储采用原子写入
- 遇到预算耗尽或重复调用时，必须返回阶段性总结

安全性：

- 默认限制在 workspace 内进行文件操作
- 高风险命令需要审批或直接阻断
- 会话恢复必须校验 workspace 一致性

兼容性：

- 支持 Windows 与非 Windows shell 环境
- 支持多 provider 接入
- 支持流式和非流式回复

### 10.2 工作区边界

1. 文件工具默认只允许操作当前 workspace。
2. 所有传入路径都会被解析并校验，禁止逃逸出 workspace。
3. 会话恢复时校验会话所属 workspace，避免跨仓库恢复。

### 10.3 Shell 风险策略

当前 Shell 风险控制是 1.0 的最小可用版本：

- 只读命令可直接执行
- 可能修改文件或环境的命令默认需要审批
- 明确高风险命令直接阻断

当前支持的审批策略：

- `always`
- `on-request`
- `never`

### 10.4 当前明确阻断的危险行为

典型阻断范围包括：

- 危险删除命令
- 破坏性磁盘命令
- 部分破坏性 Git 命令

该机制的目标不是覆盖所有安全场景，而是在 MVP 阶段提供基本可信边界。

### 10.5 已知限制

- TUI 主界面当前不提供完整工具历史侧栏
- 权限系统仍是启发式风险判断，不是规则系统
- 会话可恢复，但代码变更不可一键回退
- 输出以人类可读为主，尚未形成稳定 JSON report 协议
- 结构化计划管理不纳入当前 MVP

## 11. 产品路线

### 11.1 后续版本方向

在 MVP 版本之后，产品演进应优先考虑：

1. 更清晰的任务对象与报告对象
2. 结构化的会话内计划管理能力
3. 更细粒度的权限和信任模型
4. 更强的恢复能力
5. 更结构化的输出协议
6. 更丰富但仍受控的扩展机制
7. 更美观的 UI 界面

这些内容属于后续迭代范围，不属于当前 MVP 承诺。
