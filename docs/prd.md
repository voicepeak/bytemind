# ByteMind 产品总 PRD

## 0. 文档信息

- 产品名称：ByteMind
- 文档类型：产品 PRD
- 当前阶段：ByteMind v0.1
- 产品形态：本地 AI Coding Agent，当前主入口为 TUI
- 目标平台：macOS / Linux / Windows
- 技术实现：Go

## 1. 产品愿景与定位

### 1.1 一句话定义

ByteMind 是一个运行在本地代码仓库中的工程 Agent，当前以 Terminal / TUI 作为主交互入口，让开发者完成“理解、修改、验证、恢复

### 1.2 产品愿景

ByteMind 的长期愿景，不是做一个“更会聊天的编程助手”，而是做一个开发者愿意在真实仓库里持续使用的工程执行助手。这个助手的价值不体现在它能说多少，而体现在它能不能在本地代码仓库里理解真实问题、做出实际改动、执行必要验证，并在任务被打断后继续把事情做完。

对开发者而言，最有价值的 AI 产品不是回答一个问题，而是减少一次完整任务中的摩擦成本。ByteMind 希望减少三种最常见的摩擦：

- 为了让 AI 理解问题而不断补上下文的成本
- 在编辑器、终端、文档、测试结果之间来回切换的成本
- 任务中断后重新理解、重新定位、重新推进的成本

因此，ByteMind 面向的不是“泛泛地提供智能回答”，而是“在本地终端里帮助开发者完成真实工程任务”。

### 1.3 产品目标状态

ByteMind 的理想状态，是用户进入一个本地仓库后，能够把它当作一个可信的工程搭档来使用：

- 让它先理解项目结构和目标范围
- 让它直接修改文件或生成必要内容
- 让它执行测试、构建或脚本验证
- 让它把变更、结果和风险明确告诉用户
- 让它在中断后从正确的地方继续推进

如果这一链路可以在大多数真实任务中稳定成立，ByteMind 才算真正成为一个产品，而不仅是一个演示型 Agent。

### 1.4 产品定位

ByteMind 的定位是：`单用户、本地优先、终端优先、可控执行` 的工程 Agent。

这个定位包含四层含义：

1. 单用户  
ByteMind 当前首先服务于单个开发者在本地仓库内的工作闭环，而不是先做多人协作和团队编排。

2. 本地优先  
ByteMind 以工作区为核心边界，以本地代码、配置、命令和会话为主要上下文来源，而不是依赖云端项目索引或远程工程状态。

3. 终端优先  
ByteMind 选择 TUI 作为当前主入口，是因为目标用户本身长期工作在 Terminal 中。对于终端优先的目标用户而言，终端往往是切换成本更低、也更接近真实工程执行的位置。

4. 可控执行  
ByteMind 并不是单纯给建议，而是直接执行文件修改和命令动作。因此它必须把边界、审批、阻断、摘要、恢复这些“信任机制”作为产品本体，而不是补充特性。

### 1.5 产品不是什么

为了防止产品路线失焦，ByteMind 在当前和中期阶段都不应被定义为：

- 一个完整 IDE
- 一个团队协作平台
- 一个云端 DevOps 平台
- 一个插件市场
- 一个多 Agent 编排系统

这些方向并非永远不能做，但都不属于 ByteMind 成立所必须的前提。产品成立的第一原则，不是能力越来越多，而是主闭环越来越扎实。

## 2. 市场背景与问题定义

### 2.1 行业变化

开发者对 AI 编程产品的期待，已经从“帮我写一段代码”逐步转向“帮我把这件事做完”。过去的编程助手大多停留在解释、补全、回答问题的层面，而现实中的开发任务并不是几个离散问题的集合，而是一条连续的工程链路：要先理解仓库、定位改动点、落盘修改、执行验证、读懂结果、再继续迭代。

这意味着，真正有价值的下一代 AI Coding 产品，不应该只优化模型回答质量，而应该优化工程闭环完成质量。

### 2.2 现有方案的主要不足

现有方案并非不存在，而是普遍会在以下几个关键维度上做出取舍：

1. 上下文边界  
有些方案擅长对话和解释，但对本地工作区的真实上下文利用不够充分，用户仍然需要自己补充大量背景信息。

2. 执行深度  
有些方案能够生成建议或代码片段，但不能稳定完成文件修改、命令执行和结果判断的完整闭环。

3. 控制透明度  
有些方案具备较强执行能力，但边界、审批、追踪和恢复设计不够清晰，导致用户敢试但不敢长期依赖。

4. 工作流贴合度  
不同方案在 IDE 集成、云端协作、本地终端等形态上各有取舍，但对终端优先、本地优先的开发者来说，仍存在体验空白和分化。

### 2.3 ByteMind 要解决的核心问题

ByteMind 要解决的，不是“让 AI 更像人”，而是四个工程问题：

1. 如何让 AI 真正理解本地仓库，而不是只理解几段粘贴代码  
2. 如何让 AI 真实执行工程动作，而不是只输出建议  
3. 如何让执行过程可控、可解释、可恢复  
4. 如何让整个流程停留在开发者最自然的 Terminal 工作流中

如果这四个问题能同时成立，ByteMind 就不是一个聊天工具，而是一个真正的本地工程执行产品。

### 2.4 为什么是现在

现在做 ByteMind 有三个现实基础：

- 开发者已经形成了“让 AI 帮我直接做”的强烈预期
- 模型的理解和工具调用能力，已经足以支撑工程闭环的早期产品化
- 尽管市场上已经出现多种 Agent 方案，但在“本地工作区边界 + 真实执行 + 透明恢复”这组能力上，终端优先产品仍存在明显分化和空白

这使得 ByteMind 有机会在“足够轻”和“足够能做事”之间找到一个产品平衡点。

## 3. 目标用户与核心痛点

### 3.1 目标用户

ByteMind 的核心目标用户包括三类：

- 终端优先的后端开发者
- 多语言、跨文件工作的全栈/工具型开发者
- 关注治理、风险和效率的小团队技术负责人

这三类用户虽然工作内容不同，但他们都面对同一个问题：需要在本地仓库中连续完成多步工程任务，而现有 AI 工具通常只覆盖其中一小段。

### 3.2 用户共性痛点

三类用户的共同问题主要是：

- 项目理解成本高，AI 需要反复补上下文
- 多文件修改和命令验证过程不连贯
- 修改结果不透明，难以 review 和追踪
- 一旦任务中断，很难恢复到可继续状态
- 用户既想让 Agent 执行，又担心其越界或乱执行


### 3.3 分角色价值诉求

对小李这类终端优先的后端开发者来说，最重要的是能否在 Terminal 里完成一个完整闭环。他们希望少切换窗口，少重复解释仓库结构，少在“AI 建议”和“自己执行”之间来回跳转。

对小王这类工具型开发者来说，最重要的是多文件、多步骤、多结果的持续推进能力。他们更在意 Agent 能否真正进入现有工作流，包括快速生成、批量处理、脚本调用和结果整理。

对小黄这类技术负责人来说，最重要的是信任和治理。他们不是不想引入 Agent，而是不愿让一个边界模糊、结果不可解释、失败不可恢复的系统进入真实仓库。

### 3.4 用户采用阻力

即使产品能力本身成立，ByteMind 仍然会面对典型的采用阻力：

- 用户担心“它会改坏”
- 用户担心“它会跑错命令”
- 用户担心“它做完我看不懂”
- 用户担心“中断后我接不上”
- 用户担心“这个工具只能 Demo，不能长期用”

因此，ByteMind 的产品设计必须同时解决“能做”与“敢用”两个问题。

## 4. 核心产品价值与原则

### 4.1 核心价值

ByteMind 的核心价值不是“会话”，而是“闭环”。

对用户而言，产品价值最终体现在三件事上：

- 少解释：减少重复补充上下文
- 少切换：尽量在终端内完成理解、修改、验证
- 少返工：中断后仍能继续，而不是从头再分析

换句话说，ByteMind 不是要让用户觉得“聊得很顺”，而是要让用户觉得“事情推进得动，而且推进得稳”。

### 4.2 差异化价值

ByteMind 与通用聊天助手的区别，不在于模型本身，而在于产品组合方式：

- 它以本地工作区为主上下文，而不是以对话文本为主上下文
- 它以任务闭环为目标，而不是以回答满意度为目标
- 它把执行控制和恢复能力纳入产品本体，而不是放在外围
- 它选择终端优先，而不是依赖重型交互界面

这种差异化使 ByteMind 适合真实工程环境，也决定了它必须比普通 AI 助手更强调边界、摘要、验证和恢复。

### 4.3 产品原则

1. 本地优先  
默认围绕当前 workspace 工作，不依赖云端索引或远程工程状态。

2. TUI 优先  
主体验围绕 TUI 构建，帮助、会话、审批、状态反馈优先在 TUI 内完成。

3. 执行优先  
产品重点不是“像助手一样聊天”，而是“像工具一样完成任务”。

4. 可控优先  
文件修改、Shell 执行、路径访问必须有明确边界和确认机制。

5. 恢复优先  
任务被打断、预算耗尽、执行失败时，系统应尽量保留继续工作的基础状态。

6. 渐进增强  
先把单用户、本地、闭环做扎实，再考虑扩展能力，不通过功能堆叠制造复杂度。

### 4.4 设计取舍

ByteMind 的很多产品判断都应该服从上述原则。例如，宁可少做一个炫目的高级功能，也不要让主路径变得不可解释；宁可暂时不做多 Agent，也不要把单 Agent 的边界搞复杂；宁可输出朴素，也要保证结果清楚、可恢复、可追踪。

这意味着 ByteMind 的路线不是“功能越多越像产品”，而是“越能稳定完成主闭环，越像产品”。

## 5. 核心工作闭环
### 5.1 闭环定义

ByteMind 的核心工作闭环可以概括为四步：

理解任务 → 执行修改 → 验证结果 → 继续推进

这四步不是简单的功能罗列，而是产品成立的最小主路径。
如果用户在真实仓库中，能够沿着这条路径完成一个 bug 修复或小型改造任务，ByteMind 才算具备产品价值；如果只能完成其中某一两步，那它更像一个局部工具，而不是一个真正的工程 Agent。

### 5.2 每一步解决什么问题
1. 理解任务

系统需要围绕当前仓库和当前任务建立足够的上下文。
这一步的目标不是“读得越多越好”，而是帮助用户快速进入状态，减少反复补充背景、重复解释仓库结构和手动定位文件的成本。

2. 执行修改

系统需要基于当前任务，完成真实的工程动作，包括文件修改、命令执行和必要的操作串联。
这一步的重点不是生成一段建议代码，而是让任务真正进入“可推进”的状态。

3. 验证结果

系统需要通过测试、脚本、构建或其他方式，对修改结果进行验证。
这一步决定了产品不是“改完就算完成”，而是能够基于结果判断任务是否真的推进、是否还需要继续修正。

4. 继续推进

系统需要把本次任务的结果整理清楚，并在任务中断或未完成时，保留可继续推进的状态。
这一步的意义在于：产品不是一次性输出，而是能够承接连续任务，让用户下一次进入时不用重新从头分析。

### 5.3 为什么闭环重要

开发任务往往不是一个单点问题，而是一条连续链路。
用户真正需要的，不只是“帮我写一段代码”，而是“帮我把这件事推进下去”。

因此，ByteMind 的核心价值不在于某一个单独能力，而在于它能不能把这四步连起来：

先看懂任务
再完成修改
再验证结果
最后支持继续推进

只有这条主路径成立，ByteMind 才不是一个聊天式编程助手，而是一个可进入真实工作流的工程产品。

### 5.4 当前阶段的判断标准

在当前阶段，ByteMind 不追求覆盖所有复杂开发场景，而是先围绕最典型的真实任务验证这条闭环。
例如：

修复一个 bug
完成一个小型改造
处理一个需要多文件修改和命令验证的任务

当前阶段判断产品是否成立，不是看功能数量，而是看以下问题是否能得到正向回答：

用户是否能在本地仓库中更快进入任务状态
系统是否能真实完成修改和验证，而不只是给建议
用户是否能看懂本次任务的结果
任务中断后，是否还能低成本继续推进

## 6. 核心场景与用户故事

### 6.1 核心场景总览

ByteMind 当前应长期围绕以下场景构建，而不是追求泛化到无边界的“什么都能做”：

| 场景 | 用户目标 | 必要能力 | 优先级 |
|---|---|---|---|
| 陌生仓库理解 | 快速知道入口、结构和关键模块 | Context、Report | P0 |
| 真实问题修复 | 找到改动点、落盘修改、执行验证 | Plan、Act、Verify、Report | P0 |
| 多文件任务推进 | 跨多个文件持续处理新增或改造 | Plan、Act、Report、Resume | P0 |
| 中断后继续 | 恢复任务状态并继续推进 | Memory、Resume、Report | P0 |
| 可复用任务处理 | 复用已有技能、模板和流程 | Skill、Plan、Verify | P1 |
| 外部资源接入 | 读取仓库外资源或调用外部系统 | MCP、Policy、Approval | P1 |

### 6.2 端到端典型场景

#### 场景一：进入陌生仓库并定位修改点

用户进入一个并不熟悉的代码仓库，希望快速知道入口、配置位置和关键模块。ByteMind 需要先完成目录浏览、关键文件读取和关键词搜索，再把“项目怎么启动、主要模块在哪里、改动可能落在哪”讲清楚。这一场景验证的是 Context 和 Report 是否成立。如果没有足够依据，后续所有动作都会偏航。

#### 场景二：针对一个真实问题完成修改并验证

用户给出具体任务，例如修复测试失败、补一个配置项、改一段处理逻辑。系统需要识别任务目标、建立最小计划、修改文件、执行验证，并根据结果决定是否继续修复。

这一场景验证的是 Goal、Plan、Act、Verify 四个阶段是否可以稳定联动。

#### 场景三：任务被打断后继续推进

用户中途离开、模型预算耗尽、验证结果暂未通过，任务没有完全做完。下次再进入时，ByteMind 不应让用户从零开始，而应帮助用户快速恢复：上次改了哪些文件、跑了哪些命令、当前卡在哪里、下一步应该先做什么。恢复能力越强，产品越有持续使用价值。

### 6.3 典型用户故事

#### 小李

- 作为终端优先的后端开发者，我希望进入仓库后能快速知道入口、关键模块和改动位置，这样我不用先手工整理上下文。
- 作为终端优先的后端开发者，我希望确认方向后能直接修改代码并运行测试，这样我能在终端中完成完整闭环。
- 作为终端优先的后端开发者，我希望任务被打断后能恢复上次进度，这样我不用重新分析一遍。

#### 小王

- 作为多语言开发者，我希望能在一个任务里同时处理多个文件、脚本和配置，而不是只获得一段建议代码。
- 作为工具型开发者，我希望一次会话能持续推进“查找、修改、验证、总结”，减少上下文切换。
- 作为工具型开发者，我希望 `run` 模式能服务于快速任务和命令行调用场景。

#### 小黄

- 作为技术负责人，我希望产品默认受限于当前工作区，并对危险动作给出审批或阻断，这样团队才敢真正使用。
- 作为技术负责人，我希望每次执行都有清晰摘要，能快速 review 实际改动和验证结果。
- 作为技术负责人，我希望复杂任务跨天推进时，仍能恢复到准确上下文。

## 7. 产品边界与非目标

### 7.1 产品边界

ByteMind 的长期产品边界是：

- 以单用户工程任务为核心
- 以本地 workspace 为默认工作边界
- 以 Terminal / TUI 为主工作流入口
- 以 Agent 闭环为核心体验，而非单轮对话
- 以可扩展能力体系为增长方式，而非持续堆叠静态功能

### 7.2 当前版本边界

当前版本只是完整产品边界中的一个较小子集，重点聚焦：

- 单 Agent
- 本地仓库
- 文件读取、文件修改、Shell 执行
- 会话持久化和恢复
- 基础审批和风险阻断
- TUI 主路径闭环

以下能力属于总产品定义的一部分，但当前版本不代表已经完整交付：

- 显式的结构化 Planning
- 更强的跨次 Memory
- 用户自定义 Skills 接入
- MCP 外部能力接入
- 更稳定的结构化 Report

### 7.3 明确非目标

以下内容不属于当前产品总方向中的近期目标：

- 以图形界面为核心的完整 IDE 替代
- 多人实时协作和团队云端工作台
- 无边界自动执行的完全自治系统
- 为追求展示效果而加入的复杂多 Agent 编排
- 先做平台生态、后补主闭环的路线

需要说明的是，支持用户自定义 Skill，并不等于当前阶段要建设插件市场。Skill 解决的是能力复用与场景增强问题；插件市场涉及分发、发现、排序和生态治理，属于更后续的产品命题。

### 7.4 范围判断原则

判断一个能力是否应该进入主路线，可以看四个问题：

1. 它是否强化 `Goal / Context / Plan / Act / Verify / Report / Resume` 中的某一环  
2. 它是否显著增强用户信任、透明度或恢复能力  
3. 它是否可以通过通用机制扩展，而不是硬编码成一次性功能  
4. 它是否会明显增加复杂度，却不能提升主闭环完成率  

只有前三项有明确价值、第四项风险可控的能力，才值得进入核心路线。

## 8. 核心对象模型

### 8.1 核心对象

| 模块 | 包含对象 | 在产品中的作用 | 典型界面入口 |
|---|---|---|---|
| Session 模块 | Workspace、Session、Memory | 管理当前会话、最近会话、会话恢复和新建会话流程 | 主会话页、`/session` 模块 |
| Plan 模块 | Goal、Task、Plan、Step | 管理当前任务目标、阶段计划、当前步骤、下一步动作 | 主会话任务卡片、目标形态下的 `/plan` 面板 |
| Capability 模块 | Skill、MCP Resource | 管理可复用能力和外部资源接入能力 | 目标形态下的 `/skills` 面板、`/mcp` 面板 |
| Execution 模块 | Change、Command | 承载真实工程动作，包括文件改动和命令执行记录 | 执行记录区、变更摘要区、命令结果区 |
| Control 模块 | Policy、Approval | 约束系统行为，提供风险规则、审批确认和边界控制 | 审批弹窗、策略说明区、设置面板 |
| Result 模块 | Result、Report | 输出阶段结论、最终结果、风险说明和继续建议 | 结果卡片、任务总结页、`/report` 面板 |

这六类模块共同构成 ByteMind 的完整产品结构。用户不需要马上理解所有内部对象，能通过模块化入口快速找到“我现在在哪”、“系统做了什么”、“下一步去哪”。

### 8.2 各模块定义

#### 8.2.1 Session 模块

Session 模块负责管理用户在当前 workspace 下的会话使用流程，是当前 TUI 中最明确的一组会话入口。

其中：

- `Workspace` 表示会话所属的本地仓库目录，用于限定当前会话的归属边界。
- `Session` 表示围绕同一 workspace 持续存在的一段对话和任务上下文。
- `Memory` 表示与会话相关的摘要、恢复线索和继续推进所需信息。

在当前产品形态下，Session 模块不是以一长条对象列表暴露给用户，而是以一组会话命令组织：

- `/session`：进入会话模块入口，回到上一个会话或查看当前会话相关操作
- `/sessions [limit]`：查看最近若干个会话
- `/resume <id>`：恢复到指定 id 的对话
- `/new`：开启一个新会话

因此，从用户视角看，Session 模块的核心作用不是“理解 Session 对象本身”，而是完成四类具体动作：回到当前会话、查看最近会话、恢复历史会话、开启新会话。

在界面设计上，这一模块应先展示会话相关的一级入口，再进入具体子项，而不是把 Workspace、Session、Memory 直接作为一级菜单暴露给用户。

#### 8.2.2 Plan 模块

Plan 模块负责把用户输入转化为可推进的任务过程。

其中：

- `Goal` 表示用户真正要完成的任务目标。
- `Task` 表示系统当前推进的实际工程任务。
- `Plan` 表示围绕当前 task 形成的阶段性推进方案。
- `Step` 表示 plan 中的最小执行阶段，用于标记当前进度和中断恢复点。

在完整产品形态下，这一模块应以“当前任务 + 当前阶段 + 下一步动作”的形式出现，而不是散落在回复文本中。用户可通过 `/plan` 面板查看当前目标、已完成步骤、进行中步骤、阻塞点和下一步建议，并在需要时进入对应子项继续推进。该入口属于目标能力形态，不代表当前版本已经完整交付。

#### 8.2.3 Capability 模块

Capability 模块负责管理系统的扩展能力，是 ByteMind 区别于静态工具集合的重要组成部分。

其中：

- `Skill` 表示可复用的任务能力单元，用于沉淀某类高频任务的上下文组织方式、步骤约束、工具偏好、验证习惯和输出格式。
- `MCP Resource` 表示通过 MCP 接入的外部上下文或工具资源，用于扩展仓库外信息获取和外部系统连接能力。

在完整产品形态下，这一模块可拆分为两个入口：

- `/skills` 面板：展示当前可用 skill、已启用 skill、推荐 skill，以及用户自定义 skill。
- `/mcp` 面板：展示当前已接入的 MCP server、可用 resource、授权状态、启用状态和调用说明。

这意味着 ByteMind 的扩展能力不应只依赖官方内置功能，也应支持用户以显式方式接入自定义 skill 和自定义 MCP 资源。上述入口属于目标能力形态，不代表当前版本已经完整交付。

#### 8.2.4 Execution 模块

Execution 模块负责承载真实工程动作。

其中：

- `Change` 表示文件层面的真实改动集合，包括新增、修改、替换、补丁更新等。
- `Command` 表示执行过的命令及其输入、输出、退出状态和风险级别。

在界面上，这一模块应以执行记录区、变更摘要区、命令结果区等形式呈现，而不是让用户只能从长文本回复中回忆系统做过什么。用户应能清楚看到改了哪些文件、执行了哪些命令、哪些动作成功、哪些动作失败。

#### 8.2.5 Control 模块

Control 模块负责把系统行为约束在可信范围内。

其中：

- `Policy` 表示执行边界、风险规则和默认控制策略。
- `Approval` 表示针对修改性或高风险动作的人类确认机制。

在界面上，这一模块应以审批弹窗、边界提示、风险说明和策略设置等形式出现。用户不一定频繁主动进入该模块，但在关键风险点必须能清晰感知其存在。

#### 8.2.6 Result 模块

Result 模块负责把执行过程转化为用户可理解、可继续使用的结果。

其中：

- `Result` 表示当前阶段或最终阶段的任务结论。
- `Report` 表示面向用户的可读输出对象，用于汇总变更、执行、风险和下一步建议。

在界面上，这一模块应形成独立的结果卡片或总结页，而不是仅作为一段普通回复结束。用户需要能够快速回答四个问题：做了什么、结果如何、还有什么风险、下一步怎么继续。

### 8.3 模块关系

从产品视角看，这些模块之间存在稳定关系：

- 用户首先进入一个 `Workspace`，并创建或恢复一个 `Session`
- 在 `Session` 中，用户提出一个 `Goal`，系统据此形成当前 `Task`
- 系统围绕当前 `Task` 生成 `Plan`，并拆解为若干 `Step`
- 在任务推进过程中，系统可按需调用 `Skill`，或接入 `MCP Resource` 获取外部上下文和能力
- 实际执行过程中产生 `Change` 和 `Command`
- 所有执行动作都受到 `Policy` 和 `Approval` 约束
- 阶段性结论沉淀为 `Result`，面向用户的输出组织为 `Report`
- 关键上下文和任务状态沉淀为 `Memory`，用于下一次恢复和继续推进

因此，ByteMind 在界面上不应表现为一条线性的长流程，而应表现为围绕“当前任务”展开的多个可切换模块。用户可以从主会话进入 `/session`、`/plan`、`/skills`、`/mcp`、结果页等模块，再回到主任务继续推进。

### 8.4 界面组织原则

完整产品形态下，ByteMind 的核心模块应遵循以下界面组织原则：

- 主会话页负责承载当前任务推进，不承载所有细节配置
- `Session`、`Plan`、`Skill`、`MCP`、`Report` 等能力应通过独立面板或弹窗分组呈现
- 用户进入某一模块后，应先看到该模块的一级列表，再进入子项查看或操作，而不是一次性展示所有对象
- 所有模块都应围绕“当前任务是否更容易继续”这一目标组织，而不是围绕内部实现结构组织

## 9. 信任与控制模型

### 9.1 产品必须解决的信任问题

AI Coding 产品真正的阻力并不是“不会改”，而是“用户不敢让它改”。

因此 ByteMind 的核心信任问题包括：

- 它会不会改错地方
- 它会不会跑危险命令
- 它会不会越界访问
- 它改完后我能不能看懂
- 任务中断后我还能不能接得上
- 它接入外部资源后边界是否仍然清晰

### 9.2 最小可信模型

ByteMind 至少应具备以下最小可信能力：

- 工作区边界限制
- 危险命令阻断
- 修改性命令审批
- 变更摘要输出
- 会话恢复校验

这不是安全系统的终点，但必须是产品的起点。

### 9.3 控制域划分

| 控制域 | 主要风险 | 基本控制方式 |
|---|---|---|
| 文件系统 | 越界读写、误删、误改 | 路径校验、workspace 边界、危险操作限制 |
| Shell | 错误命令、破坏性执行 | 风险识别、审批、阻断、执行记录 |
| Git | 不可逆回退、历史破坏 | 破坏性命令限制、审批、明确摘要 |
| 网络 | 发送敏感信息、调用未知服务 | 默认受限、显式开关、来源可见 |
| MCP | 外部资源越权、结果不可解释 | 资源级授权、调用可追踪、最小权限 |
| 会话恢复 | 恢复失真、上下文错配 | workspace 校验、恢复摘要、继续建议 |

### 9.4 风险等级与处理原则

ByteMind 的动作应至少分为四类风险级别：

| 风险级别 | 典型动作 | 默认处理方式 |
|---|---|---|
| L0 | 只读浏览、读取文件、搜索 | 可直接执行 |
| L1 | 可逆或低风险动作 | 允许执行，但应可追踪 |
| L2 | 修改性动作 | 需要明确策略约束或用户确认 |
| L3 | 高破坏性或不可逆动作 | 默认阻断或极强确认 |

风险判断不应只看命令字面，还应结合上下文、路径和目标对象。

### 9.5 审批、审计与恢复

ByteMind 的控制模型不应只停留在“执行前问一句”。一个可信 Agent 至少需要：

- 审批前：告诉用户要做什么、为什么做、风险在哪里
- 执行中：清楚展示当前阶段和关键动作
- 执行后：提供变更摘要、命令摘要、结论和下一步建议
- 中断后：提供可继续的恢复入口，而不只是历史日志

### 9.6 用户信任的建立路径

用户对 Agent 的信任通常分三个阶段：

1. 愿意试用  
把它放进一个真实仓库里试一下。

2. 愿意交任务  
让它执行真实修改、验证和继续修复。

3. 愿意依赖  
在日常工作和跨天任务中重复使用它。

ByteMind 的控制设计应服务于这三个阶段的推进，而不是一开始就要求用户完全放权。

### 9.7 信任设计原则

1. 默认受限  
默认只在当前 workspace 内工作。

2. 默认透明  
执行过什么、改动了什么、验证了什么，应能被用户理解。

3. 默认可中断  
用户应能在关键风险点阻止系统继续执行。

4. 默认可恢复  
系统失败时，应优先保证可继续，而不是优先追求“自动做完”。

### 9.8 对产品设计的要求

这意味着 ByteMind 的产品设计不应一开始就要求用户完全放权，而应让用户沿着“试用 -> 交任务 -> 依赖”的路径逐步增加信任。

对应地：

- 前期重点是边界清晰、反馈可见、风险可控
- 中期重点是结果稳定、过程透明、恢复可继续
- 后期重点是高频任务中的持续可用性和重复使用习惯

## 10. 产品能力架构

### 10.1 能力分层

从产品视角看，ByteMind 的能力应稳定划分为九层：

| 层级 | 核心职责 | 用户可感知结果 | 当前阶段 |
|---|---|---|---|
| 交互层 | TUI、CLI、消息输入、阶段反馈、审批交互 | 用户知道系统在做什么 | 已进入主线 |
| 上下文层 | Workspace 识别、目录浏览、文件读取、文本搜索、历史恢复 | 系统能基于真实依据工作 | 已进入主线 |
| 规划层 | 任务拆解、阶段推进、下一步判断 | 任务不是盲走，而是有路径推进 | 规划中 |
| 执行层 | 文件修改、命令执行、外部调用、结果捕获 | 系统能完成真实工程动作 | 已进入主线 |
| 记忆层 | 摘要沉淀、任务状态、恢复快照、关键约束保留 | 任务可跨轮、跨次继续 | 规划中 |
| 技能层 | 可复用任务模板、场景指令、执行习惯、验证模式 | 常见任务更稳定、更少重复配置 | 规划中 |
| 扩展层 | 通过 MCP 接入外部资源和工具 | Agent 不只局限于本地静态工具 | 规划中 |
| 控制层 | 权限边界、风险识别、审批、阻断、预算保护 | 系统可控、可审查 | 已进入主线 |
| 结果层 | 变更摘要、执行摘要、阶段结论、继续建议 | 用户能理解结果并继续使用 | 需增强 |

### 10.2 各层的长期要求

- 交互层  
目标不是视觉复杂，而是让用户始终知道当前阶段、关键动作和下一步入口。

- 上下文层  
目标不是收集越多越好，而是为当前任务建立足够依据。

- 规划层  
目标不是生成长计划文案，而是把任务推进变得稳定、可中断、可继续。

- 执行层  
目标不是展示能调用多少工具，而是完成高频工程动作。

- 记忆层  
目标不是堆积日志，而是沉淀继续任务真正需要的信息。

- 技能层  
目标不是做一个花哨市场，而是把高频场景的最佳实践稳定复用。

- 扩展层  
目标不是接越多外部系统越好，而是在边界清晰的前提下引入必要外部能力。

- 控制层  
目标不是让系统什么都做不了，而是在能做事与敢用之间取得平衡。

- 结果层  
目标不是输出更多文本，而是形成可 review、可恢复、可衔接的结果对象。

### 10.3 架构演进约束

从产品和架构共同视角看，后续演进应遵守几个约束：

- 不让交互层绑死底层能力，避免未来无法扩展
- 不让执行层绕开控制层，避免风险失控
- 不让结果层只是原始日志堆积，必须能形成可读摘要
- 不让会话恢复只是恢复文本历史，而要恢复任务上下文

这个能力分层既适用于产品规划，也适用于后续技术拆分。

## 11. 关键能力定义

### 11.1 Planning

Planning 是把任务从“我知道要做什么”转化成“我知道下一步做什么”的能力。

它的产品要求包括：

- 能把目标拆成可执行阶段，而不是只给长篇解释
- 能根据验证结果动态调整后续动作
- 能在中断后恢复到某个明确步骤，而不是重新分析整件事

Planning 的最小产品形态不是复杂项目管理系统，而是：

- 当前任务状态
- 当前步骤与下一步
- 阶段性完成判断
- 可继续的简明计划摘要

### 11.2 Memory

Memory 不是日志堆积，而是任务继续所需信息的主动沉淀。

ByteMind 的 Memory 至少应包含：

- 任务目标和约束
- 关键上下文结论
- 已产生的主要变更
- 已执行的重要命令和结果
- 当前卡点与下一步建议

### 11.3 Skill

Skill 是可复用的任务能力单元，用于把某一类高频任务的上下文、步骤、工具偏好、验证习惯和输出格式固化下来。

Skill 的价值不在于“给系统更多指令”，而在于：

- 降低重复配置成本
- 提高特定场景的成功率和一致性
- 让团队或个人能够积累稳定工作方式

ByteMind 支持用户自定义 Skill，并不等于当前阶段要建设插件市场。Skill 是本地可接入的能力包机制，解决的是能力复用与场景增强问题；插件市场则涉及分发、发现和生态治理，属于更后续的产品议题。

### 11.4 MCP

MCP 用于把 ByteMind 从“只能看当前仓库”扩展为“可以在边界清晰的前提下读取外部上下文或调用外部工具”的 Agent。

MCP 的价值主要体现在：

- 获取仓库外文档、任务单、接口说明、环境信息等上下文
- 连接已有研发系统，而不是在产品内部重复建设
- 让 Agent 的能力扩展具备统一接入模型，而不是不断新增专有接口

MCP 的产品前提不是“先接很多源”，而是：

- 资源来源可见
- 权限边界明确
- 调用过程可追踪
- 结果可解释

### 11.5 Report

Report 是面向用户的稳定结果对象，不应等同于一段随意的助手回复。

一个可用的 Report 至少应回答：

- 这次任务想做什么
- 实际改了什么
- 执行了什么
- 验证结果如何
- 风险或未完成项是什么
- 下一步建议是什么

长期来看，Report 应同时支持：

- 人类可读
- 稳定字段
- 恢复继续
- 对接后续流程

### 11.6 Human-in-the-loop

ByteMind 的目标不是完全移除人，而是把人放在最有价值的位置：

- 在目标和边界不清晰时确认方向
- 在修改性或高风险动作前做审批
- 在结果不明确时做判断和取舍
- 在复杂任务中决定是否继续、回退或切换路径

一个真正可用的 Agent 产品，不是“彻底自动化”，而是“让用户用更少精力维持更强控制”。

## 12. 版本策略与产品路线

### 12.1 路线原则

ByteMind 后续不应沿着“加越来越多功能”演进，而应沿着“把工程闭环做得越来越成熟”演进。

路线判断原则是：

- 是否让任务完成更稳定
- 是否让执行更可信
- 是否让恢复更自然
- 是否让扩展能力进入统一机制，而不是新增孤立功能

### 12.2 阶段路线

| 阶段 | 目标 | 关键能力 | 判断标准 |
|---|---|---|---|
| 阶段一：单 Agent 本地闭环 | 在真实仓库中完成最小可用闭环 | Context、Act、Verify、基础 Report、基础 Resume | 用户可完成真实任务且结果可解释 |
| 阶段二：结构化推进 | 让任务推进更稳定 | 显式 Planning、阶段状态、结果更清晰 | 复杂任务不再频繁盲走 |
| 阶段三：可恢复闭环 | 让任务跨轮、跨次更自然 | 增强 Memory、恢复快照、继续建议 | 中断后可继续率明显提高 |
| 阶段四：可复用能力 | 让高频任务稳定复用 | Skill、模板化最佳实践、场景化验证 | 常见任务成功率和一致性提升 |
| 阶段五：可扩展 Agent | 让系统安全接入外部能力 | MCP、外部资源授权、扩展控制 | 外部上下文和工具接入变得统一且可控 |
| 阶段六：更成熟协同 | 进入更复杂任务与协同模式 | 更细粒度 Policy、更强 Report、可选协同机制 | 在不失控的前提下支持更复杂工作流 |

### 12.3 暂缓项

以下方向应谨慎延后，避免干扰主线：

- 以展示效果为导向的复杂多 Agent 编排
- 多人实时协作和云端平台能力
- 插件市场先于能力模型
- 复杂可视化界面先于闭环成熟
- 大而全但不服务主闭环的规则系统

这些能力本身不一定没有价值，但在主闭环尚未完全稳定前，它们更可能增加复杂度，而不是增加产品价值。

### 12.4 进入下一阶段的判断标准

产品是否进入下一阶段，不应只看“功能已经开发”，而应看以下问题：

- 常见任务是否已经有可接受的完成率
- 用户是否开始形成重复使用习惯
- 风险与恢复机制是否已足以支撑真实仓库使用
- 新阶段的能力是否建立在上一阶段成熟基础上

只有这些条件成立，路线推进才有意义。

## 13. 核心指标

### 13.1 产品指标

- 含工具调用任务的完成率
- 含文件修改任务的完成率
- 含命令执行任务的完成率
- 会话恢复后的继续完成率
- 用户在一次会话中完成闭环的比例

这些指标反映的是产品是不是“真的能把事情做完”，而不是只是“生成过一些内容”。

### 13.2 信任指标

- 高风险命令阻断成功率
- 审批后继续完成任务的比例
- 用户对结果摘要可理解性的主观评分
- 因边界错误或越界操作导致的失败率

信任指标的重要性不低于完成率。对于 ByteMind 这类产品来说，用户不敢交任务，本质上和产品做不完任务一样致命。

### 13.3 效率指标

- 首次可交互时间
- 首次有效结果时间
- 单任务平均上下文往返次数
- 用户为完成任务而离开终端的比例

效率指标反映的是 ByteMind 是否真正降低了工程摩擦。如果用户仍然要频繁离开终端、重新解释上下文、手工整理结果，那么产品价值就没有真正成立。

### 13.4 指标使用原则

ByteMind 的指标不应被孤立看待。例如：

- 高完成率如果建立在高风险误执行上，没有意义
- 高安全性如果建立在几乎无法执行真实任务上，也没有意义
- 高摘要质量如果任务本身做不完，同样不能证明产品成立

因此，产品评估应始终把“完成率、信任度、效率”三类指标结合起来看。

## 14. 风险与挑战

### 14.1 产品风险

- 用户会把它误解成聊天工具，而不是工程工具
- 任务一旦不稳定，用户很难形成持续使用习惯
- 功能扩张过快会稀释主路径，导致产品失焦

### 14.2 技术风险

- 不同仓库结构差异大，理解成本高
- 多步执行容易在局部决策中偏航
- Shell 风险识别天然存在边界
- 会话恢复容易出现“看似恢复，实则失真”

### 14.3 组织与落地风险

- 团队负责人可能担心权限和治理问题
- 不同开发者对终端、模型、配置环境的要求不一致
- 如果产品输出不够清晰，review 和协作链路很难接住

### 14.4 应对策略

- 优先收敛主路径，不以扩功能解决核心问题
- 用清晰摘要和状态反馈提高可理解性
- 用边界、审批、阻断控制高风险动作
- 用恢复机制和阶段总结降低中断成本
- 用版本文档和产品文档分层，防止产品定位再次混乱

## 15. 结论

ByteMind 作为产品要成立，关键不在于它能支持多少新功能，而在于它能否把 “本地仓库中的工程执行闭环” 做成一个可信、可恢复、可持续使用的终端产品。

真正需要长期坚持的主线只有一条：

在不显著增加产品复杂度的前提下，让用户越来越稳定地完成理解、修改、验证、恢复这件事。

如果这一主线被持续守住，ByteMind 就会从一个早期本地 Agent，逐步成长为开发者在真实仓库里愿意反复使用、甚至愿意依赖的工程执行产品。
